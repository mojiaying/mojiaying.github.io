                    并行：多个异步请求同时进行
                    Promise.all可以保证，promises数组中所有promise对象都达到resolve状态，才执行then回调。

                    let promise = () =>{
  return [1000, 2000, 3000].map(cur => {
    return new Promise((resolve, reject)=>{
      setTimeout(()=> {
        console.log(`after ${cur}ms`)
        resolve(`${cur}ms`)
      }, cur)
    })
  })
}
Promise.all(promise()).then((res) => {
  console.log('返回结果：',res)
  console.log('end-----------')
})

                    Promise.all 并发限制 如果你的promises数组中每个对象都是http请求，而这样的对象有几十万个。

                    那么会出现的情况是，你在瞬间发出几十万个http请求，这样很有可能导致堆积了无数调用栈导致内存溢出。

                    方法一：

                    1. 初始连续调用最大并非量个请求
                    2. 每回来一个结果，把放到对应的数组位置，并且发下一个请求
                    3. 直到全部都调完，并且所以结果数组每个位置都有东西，就把结果数组resolve出去
                    function multiRequest(promises, max){
  return new Promise(resolve => {
    let len = promises.length // 请求总数
    let cur = 0 // 当前数
    let resArr = new Array(len).fill(false)
    for(let i = 0; i< max;i++){
      next(cur++)
    }
    function next(index) {
      console.log(`开始发第${index}次`)
      promises[index]().then(res => {
          callback(res)
        }).catch(err => {
          callback(err)
        })
        function callback(res) {
          console.log(`第${index}----------------------------------次请求返回`)
          resArr[index]= res
          if (cur >= len) {
            console.log('所有请求已发完-----------')
            if(!resArr.includes(false)) {
              console.log('所有结构接收完毕返回结果数组-----------', index)
              resolve(resArr)
            }
          } else {
            next(cur++)
          }
        }
    }
  })
}

                    方法二：
                    使用Promise.all做并发限制。每个时刻并发执行的promise数量是固定的，最终的执行结果还是保持与原来的Promise.all一致。 以批量下载文件为例

                    send = () => {
  const arrList = Array.from({length:100}, (v,k) => k); // 这边假设生成勾选的100条数据
  const limit = 5; // 并发请求数量
  // 请求下载文件请求和操作
  const downloadMethond = (url) => {
    // 单个文件请求完处理
    // some code
  }
  // 将所有数据定义成一个数组，这个数组的每一个元素返回一个函数，这个函数里面返回一个异步下载
  const asyncFuncList = arrList.map((item) => {
    return () => {
      return downloadMethond(item).then((res) => {
        const next = asyncFuncList.shift(); // 队列中还有等待，则取出等待的第一个
        if (next) {
          next(); // 执行这个
        } else {
          console.log('全部执行完成')
        }
      })
    }
  });

  const promiseList = [];
  // 实现并发请求
  for (let i = 0; i < limit; i ++) {
    const promiseItemFun = asyncFuncList.shift();
    promiseList.push(promiseItemFun());
  }

  Promise.all(promiseList).then((res) => {
    // 这边可以做一些全部请求完的操作
  })
}

                    1. tcp和udp
                    Details

                    都是传输层协议 TCP面向连接，需要三次握手建立连接，有状态，精准记录发送的数据，接收的数据，没有接收的数据，而且保证数据包按序到到达，意识到丢包会控制发送速度或者重发 UDP面向无连接，没有连接过程

                    1. http
                    Details

                    HTTP 请求过程：

                    • DNS：应用层协议，负责域名解析，通过域名来获取目标主机的 IP
                    • TCP：传输层协议，提供可靠的字节流服务，需要三次握手、四次挥手
                    • IP：网际协议，把各种数据包传输给对方，需要有：IP地址和MAC 地址
                    • ARP：网络层协议，是解析地址的协议，凭借 MAC 地址进行通信，根据通信方IP反查出对应的 MAC 地址
                    1. vue生命周期
                    Details

                    vue2:

                    1. beforeCreate( 创建前 )：此时组件的选项对象还未创建，el 和 data 并未初始化
                    2. created ( 创建后 ）：实例已完成以下配置：数据观测、属性和方法的运算，watch/event事件回调，完成了data 数据的初始化，el没有
                    3. beforeMount：完成了el和data 初始化，注意此时还没有挂在html到页面上。
                    4. mounted:模版渲染完成，可以获取DOM元素
                    5. beforeUpdate:在数据更新之前被调用，发生在虚拟DOM重新渲染和打补丁之前，可以在该钩子中进一步地更改状态，不会触发附加地重渲染过程
                    6. updated（更新后）:在由于数据更改导致地虚拟DOM重新渲染和打补丁只会调用，调用时，组件DOM已经更新，所以可以执行依赖于DOM的操作，然后在大多是情况下，应该避免在此期间更改状态，因为这可能会导致更新无限循环，该钩子在服务器端渲染期间不被调用
                    7. beforeDestroy（销毁前）:实例仍然完全可用,一般在这一步做一些重置的操作，比如清除掉组件中的定时器 和 监听的dom事件
                    8. destroyed（销毁后）:在实例销毁之后调用，调用后，所以的事件监听器会被移出，所有的子实例也会被销毁，该钩子在服务器端渲染期间不被调用

                    1. 关系型数据库的范式
                    Details

                    数据库范式有6种：第一范式（1NF）、第二范式（2NF）、第三范式（3NF）、BC范式（BCNF）、第四范式（4NF）、第五范式（5NF）

                    1. 第一范式（1NF）：要求每一列都是不可分割的原子数据项。
                    2. 第二范式（2NF）：消除非主属性对主键的部分依赖（不能只与主键的某一部分相关）
                    3. 第三范式（3NF）：消除非主属性对主键的传递依赖（必须每一列都和主键直接相关，非主属性不依赖其他非主属性）
                    4. BC范式（BCNF）：消除主键内的传递关系（一个主键不能直接依赖另外一个主键）
                    5. 第四范式（3NF）：消除表中的多值依赖
                    6. 第五范式：（消除非候选码的表字段连接依赖）
